这个方向非常适合，而且比“泛科研助教”更强，因为它同时满足三件事：

第一，它有**清晰的学科边界**：城市治理、城市规划、城市安全。
第二，它有**明确的论文来源策略**：以 *Cities* 为主，再逐步扩到少量补充期刊。
第三，它非常适合作为您这 5 周课程的主案例，因为能自然承载“最小智能体 → 工具调用 → LangChain → LangGraph → RAG → 评估”的主线。您前面已经定下，这 5 周必须讲透的是智能体范式、架构、提示工程、工具调用、记忆与 RAG 融合、推理规划、评估，而 LangChain/LangGraph 是工程实现载体。

另外，*Cities* 的官方定位是“**The International Journal of Urban Policy and Planning**”，重点在城市规划与政策；其 aims & scope 强调分析和评估城市发展与管理，并促进更有效的城市政策实施。最近卷期里也已经明确出现了与您方向高度贴合的主题和文章，例如 *Artificial Intelligence in urban design: A systematic review*、*Predicting urban vitality and pedestrian road safety in urban areas based on machine learning*，以及围绕地方政府 AI 的综述性研究。这说明您的选题如果想贴近 *Cities*，**标题和系统定位都应以城市问题为中心，以 AI/Agent 为方法支撑，而不是反过来把“算法大杂烩”放在最前面**。([ScienceDirect][1])

# 八、Codex 在这里怎么用，才最合理

Codex 不是替您“直接生成一个完整 agent”，而是作为**分模块开发协作者**。
OpenAI 官方对 Codex 的定位是：它是一个软件开发 coding agent，可以帮助编写代码、理解代码库、修复 bug，并在本地工具或云环境中协助开发；官方还提供了 app、IDE 扩展、skills 和 MCP 接入等能力。([OpenAI][2])

所以您最稳的方式不是一句“帮我做个科研助教”，而是把任务拆成小模块交给 Codex。

## 建议的开发原则

每个 `.py` 文件只做一件核心事，并要求：

* 中文注释
* 文件头部写清“本模块做什么”
* 函数注释写清输入/输出
* 必要异常处理
* 测试用例或 demo 函数
* README 标明依赖关系

## 建议的仓库结构

```text
cityscholar-agent/
│
├─ app.py                     # 程序入口，负责启动主流程
├─ config.py                  # 配置文件，存放路径、模型参数、检索参数
├─ requirements.txt           # 依赖清单
├─ README.md                  # 项目说明
│
├─ data/
│   ├─ raw_papers/            # 原始PDF论文
│   ├─ metadata/              # 论文元数据
│   └─ index/                 # 向量索引或本地数据库
│
├─ core/
│   ├─ agent.py               # 智能体主逻辑
│   ├─ state.py               # 状态管理
│   ├─ planner.py             # 任务拆解与简单规划
│   └─ prompts.py             # 系统提示词模板
│
├─ rag/
│   ├─ loader.py              # PDF加载与文本抽取
│   ├─ splitter.py            # 文本切块
│   ├─ embedder.py            # 向量化
│   ├─ retriever.py           # 检索器
│   ├─ reranker.py            # 重排序（V2再做）
│   └─ knowledge_base.py      # 知识库管理
│
├─ tools/
│   ├─ search_tool.py         # 按主题/元数据检索论文
│   ├─ compare_tool.py        # 多论文比较
│   ├─ outline_tool.py        # 综述提纲生成
│   └─ export_tool.py         # Markdown导出
│
├─ eval/
│   ├─ eval_retrieval.py      # 检索效果评估
│   ├─ eval_answer.py         # 回答质量评估
│   └─ test_queries.json      # 测试问题集
│
└─ notebooks/
    ├─ week1_min_agent.ipynb
    ├─ week2_tools.ipynb
    ├─ week3_langgraph.ipynb
    ├─ week4_rag.ipynb
    └─ week5_eval.ipynb
```

这套结构很适合让 Codex 一块一块写。

# 九、给 Codex 的任务应该怎么下

## 给 Codex 的通用要求

1. 使用 Python 编写，可在本地运行。
